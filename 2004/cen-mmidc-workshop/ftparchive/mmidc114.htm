This is the Draft CWA under the 2004 Workplan of the CEN/ISSS Workshop on Dublin Core Metadata for Multimedia Information - Dublin Core (MMI-DC) of the European Committee for Standardization CEN. This CWA is a revised version of CWA 13988, Guidance information for the use of Dublin Core in Europe, which was last updated in April 2003. This work aims at a complete overhaul of the guidance information based on various sources that were published over the last years and acts as such as an index and summary of guidance material.

The Draft CWA is submitted to the MMI-DC workshop for public comment in July 2004. Following discussion at the MMI-DC meeting in September 2004, the Draft CWA will be finalised for publication in December 2004.

Introduction

This document is the result of work carried out under the work plan of the CEN MMI-DC Workshop during the year 2004.

Previous work was done in this area by the MMI-DC workshop itself, which led to the publication of "Guidance information for the use of Dublin Core in Europe" (CWA 13988-2003 [1]) and by volunteers in the DCMI User Documentation Guide Working Group, which resulted in the DCMI User Guide "Using Dublin Core" [2].

A need was identified to revise CWA13988-2003 taking into account recent developments in the Dublin Core specifications and providing a roadmap to existing documentation.

1 Scope

The current document is a further development of CWA13988-2003. One of the issues that were identified with this earlier CWA was that it duplicated a great deal of information that was also available from the Web site of the Dublin Core Metadata Initiative (DCMI). At the time that the previous CWA was written, it did make sense to bring all information physically together in one place so that people who were considering implementing metadata systems based on Dublin Core had a comprehensive overview of all elements and other terms. In addition, the CWA contained a list of projects and descriptions of good-practice cases that would help implementers understand better what issues they needed to consider in implementing Dublin Core metadata, at the same time giving them access to people and projects that they could get in contact with if they so desired. However, the information contained in CWA 13988-2003 has gone out of date because of further developments within DCMI and the fact that projects have terminated.

The current document therefore uses a different approach. Wherever possible information is not copied from other sources but included in the form of pointers with a short summary of the scope and content of the information that can be found following the pointer. When changes occur in the source material, the pointers will automatically point to the latest version of the documentation. The persistence policy of DCMI will ensure the long-term availability of documents.

Much of the current document can be seen as a roadmap to DCMI documentation, including the semantic and technical specifications and recommendations, as well as the organisation structure and procedures of DCMI itself.

The purpose of the information presented here is to provide an overview of documentation and answers to a number of questions that implementers may have.

 

 

 

 

2 Dublin Core metadata resources

2.1 Model and principles

The early years of the Dublin Core Metadata Initiative coincided with the rise of the World Wide Web Consortium (W3C), and the development of a data model for use with Dublin Core was heavily influenced by -- and in turn influenced -- the development of a data model for W3C's Resource Description Framework (RDF). The basic data model -- that of a single Resource with Properties (Elements), along with Element Refinements and Encoding Schemes to support the creation of semantically more specific metadata within the limits of the Dumb-Down Principle – is described in a two-page document, "DCMI Grammatical Principles" [1].

A need to articulate the assumptions and policies underpinning the identification of DCMI metadata terms with Uniform Resource Identifiers (URIs) and the maintenance of those URI-identified terms over time led in 2001 to the formulation of a "DCMI Namespace Policy" [2]. Technical considerations in the formation of names for metadata terms are described in a "DCMI Policy on Naming Terms" [3].

As Dublin Core came to be deployed in a diversity of application environments, the need emerged for a data-model framework against which the various contexts for Dublin Core could be compared and mapped for the purposes of interoperability. This work has resulted in a draft "DCMI Abstract Model" [4]. It is expected that the Abstract Model will eventually replace "DCMI Grammatical Principles" as the point of reference for Dublin Core metadata applications.

2.2 Metadata Terms: "Semantic Recommendations"

The original fifteen-element Dublin Core was developed at a series of workshops starting in 1995 and has remained essentially unchanged since the publication of the Dublin Core Metadata Element Set Version 1.1 in July 1999. DCMES Version 1.1 has also been published by the standardization agencies IETF, NISO, and ISO [1].

Since 2000, the Dublin Core Metadata Initiative has also maintained a growing set of "qualifiers", additional elements, and a controlled vocabulary of values for use with the Type element. The terminology used to describe these terms has evolved over the years in conjunction with the clarification of a data model for Dublin Core metadata (see Section 3.2). In order to minimize confusion, DCMI now publishes the complete set of metadata terms of all categories -- including the fifteen terms of DCMES Version 1.1 -- in one consolidated document called "DCMI Metadata Terms" [2]. Documentation on DCMI metadata terms is published not just as Web documents but also in the form of RDF and XML schemas [3].

2.3 Technical Specifications

Along with the Semantic Recommendations and the Namespace Policy, the Web page of key DCMI documents [1] includes several important specifications for encoding Dublin Core metadata using different implementation technologies.

"Expressing Dublin Core in HTML/XHTML meta and link elements" describes how a Dublin Core metadata record can be embedded within a Web document using special tags [2].

 

"Guidelines for implementing Dublin Core in XML" recommends how to create metadata records using W3C's XML Schemas language [3].

Implementers who wish to express Dublin Core metadata in RDF have a choice of two approaches. The simpler approach, outlined in "Expressing Simple Dublin Core in RDF/XML", specifies an encoding for records using the fifteen Dublin Core elements with string values and no additional elements or qualifiers and describes how to point to the resulting records from Web pages [4].

More sophisticated from a modelling standpoint is the specification "Expressing Qualified Dublin Core in RDF/XML", a "Proposed DCMI Recommendation" potentially subject to further revision [5].

2.4 CEN Workshop Agreements

As part of its Workshop on Metadata for Multimedia Information, the European Committee for Standardization (CEN) in Brussels has for several years hosted an international workshop, which produces guidelines and specifications related to Dublin Core [1]. The CEN workshop maintains a formal liaison to DCMI with which it closely cooperates on the review and revision of CEN Workshop Agreements (CWAs).

CWA 14855, "Dublin Core Application Profile Guidelines", specifies a format for documenting which metadata terms an organization, information provider, or user community uses in its metadata as a step towards automating semantic interoperability across a broad diversity of information sources [2].

CWA 14857, "Mapping between Dublin Core and ISO 19115", defines a mapping between Dublin Core and a key international standard for describing geographic information and services [3]. CWA 14856 situates this crosswalk in the broader context of mappings among standards for describing geographic information generally [4]. CWA 14858 recommends a selection of metadata terms from Dublin Core and from geospatially oriented namespaces as a basis for improving the discovery of geographical information in cross-domain searches [5].

CWA 14859, "Guidance on the use of Metadata in eGovernment", surveys national and international efforts to converge on standards and guidelines for describing government information, offering a roadmap for agencies wishing to undertake work in this area [6]. CWA 14860, "Dublin Core eGovernment Application Profiles", looks in more detail at differences between national metadata profiles based on Dublin Core and proposes a synthesis of their salient features into one overarching profile [7].

2.5 Other resources

In addition to developing Semantic Recommendations, technical specifications, and metadata models, DCMI points to or maintains other related resources of use to metadata implementers [1]. AskDCMI, modelled on reference librarian services, provides a Web form where implementers can pose basic questions on the use and application of Dublin Core [2].

"Using Dublin Core" is a concise, readable, high-level overview of the Dublin Core philosophy and practice [3]. It includes a separately maintained glossary of Dublin-Core-related concepts [4], and a historical bibliography of articles and books related to Dublin Core [5].

The DCMI Registry, a database designed to promote the discovery and reuse of existing metadata definitions, is significant in several respects:

  • as a cross-referenced set of translations of Dublin Core into over twenty languages;
  • as an application based on Semantic Web technology such as RDF; and
  • as an open-source software development by the OCLC Office of Research; and as a prototype for distributed registry services based on shared models, with compatible registry installations in Germany, Japan, and China [6].

DCMI maintains Web pages pointing users to translations of DCMI specifications into other languages [7], available training materials and courseware about metadata [8], projects related to or using Dublin Core [9], and available software for creating, extracting, editing, or converting Dublin Core metadata [10].

General policies regarding the creation, use, and dissemination of DCMI outputs are described in the "DCMI Publication Policy" [11] and "DCMI Intellectual Property Notice and Legal Disclaimers" [12].

3 Dublin Core Metadata Initiative

3.1 Mission and scope

The Dublin Core Metadata Initiative (DCMI) is an organization dedicated to promoting the widespread adoption of interoperable metadata standards and developing specialized metadata vocabularies for describing resources that enable more intelligent information discovery systems.

The mission of DCMI is to make it easier to find resources using the Internet through the following activities:

  1. Developing metadata standards for discovery across domains,
  2. Defining frameworks for the interoperation of metadata sets, and,
  3. Facilitating the development of community- or disciplinary-specific metadata sets that are consistent with items 1 and 2

The range of activities of DCMI includes:

  • Standards development and maintenance, such as organizing international workshops and working group meetings directed toward developing and maintaining DCMI recommendations.
  • Tools, services, and infrastructure, including the DCMI metadata registry to support the management and maintenance of DCMI metadata in multiple languages.
  • Educational outreach and community liaison, including developing and distributing educational and training resources, consulting, and coordinating activities within and between other metadata communities.

See: http://dublincore.org/about/.

3.2 Organisational structure and governance

DCMI Board of Trustees

The Board of Trustees oversees the activities of the DCMI and its Directorate. It advises the Directorate on strategic issues and allocation of financial resources, contributes to the promotion of the Initiative through liaisons with the public and private sectors and assists in securing support for the Initiative. The current Board members are based in various parts of the world (North America, Europe, Asia, Australia) and have a strong mix of backgrounds (libraries, education, information science, commerce, consultancy, international organizations and foundations). The Board of Trustees generally meets face-to-face twice yearly and additionally via teleconferences as appropriate.

DCMI Host

The DCMI Host shares major administrative responsibilities and employee costs of DCMI. These include personnel administration, legal costs, and infrastructure support (including servers, associated software, and Web site hosting, maintenance and development). The current host of DCMI is OCLC Online Computer Library Center Inc., in Dublin, Ohio, USA.

DCMI Affiliates

DCMI Affiliates are national or regional organizations that represent stakeholder communities within their region. Affiliates are established on a contractual basis and have rights and responsibilities as designated in an Affiliate Agreement executed between the Affiliate and DCMI (or its Host).

DCMI Directorate

The DCMI Directorate is responsible for the day-to-day management of DCMI and its activities. DCMI staff work under responsibility of the Directorate.

DCMI Usage Board

The mission of the Usage Board is to ensure an orderly evolution of the metadata terms maintained by the Dublin Core Metadata Initiative. The Usage Board evaluates proposals for new terms (or changes to existing terms) in light of grammatical principle, semantic clarity, usefulness, and overlap with existing terms. To proposals that are accepted, it assigns a specific status. The Usage Board also evaluates constructs that use DCMI terms, such as Application Profiles.

The UB consists of at least seven and no more than eleven people appointed by the DCMI Directorate. Appointments to the Usage Board are based on demonstrated longstanding commitment to DCMI principles and general expertise in the resource description arena. Appointments are for three years, and the Directorate strives to balance the geographic composition of the Usage Board.

DCMI Advisory Board

The DCMI Advisory Board gives advice to the DCMI Directorate on all technical and strategic issues that occur during the operation of the DCMI. The Advisory Board has a dual role in the DCMI: an internal role to assist in and advise on the developments that take place within DCMI, and an external role to liaise with the stakeholder community and other global metadata initiatives. It has an important role in reviewing charters and activities of Working Groups. This role puts specific emphasis on recognizing and signalling potential conflicts between activities, suggesting specific coordination between Working and Interest Groups, and proposing specific contacts with individuals, groups, or organizations outside the DCMI.

The Advisory Board is comprised of: Working Group Chairs; official liaisons to other metadata activities; representatives of stakeholder organizations, regional centres and communities; and invited experts appointed by the DCMI Directorate.

3.3 Operational principles and work structure

Operational Principles

The diversity of DCMI participants is great, spanning more than 50 countries, and representing many sectors, including governments, libraries, museums, archives, commercial enterprises, research and education communities, and more. Finding common ground and consensus amid such diversity is an ongoing challenge. DCMI has, since its inception, retained the interest of such a community through adherence to the principles of operation enumerated here.

Open consensus building. DCMI is open to participation to all interested parties and stakeholders. Any individual or group with expertise in metadata is welcome to participate in the DCMI community by contributing ideas through joining working groups, subscribing to DCMI mailing lists, and attending working group or plenary conferences.

International Scope and participation. DCMI is founded on a commitment to international, multilingual representation of its standards. This commitment is exemplified by the wide array of translations of DCMI standards and documentations, while the Board of Trustees and the Advisory Board each have diverse representation from many countries.

Neutrality of purposes and business models. DCMI is neutral as to the purposes for which DCMI metadata standards might be used, and encourages the adoption of these standards in the public and private sectors.

Neutrality of technology. DCMI standards are fundamentally concerned with semantics – the meaning of assertions about information resources. DCMI attempts to facilitate the expression of these semantics in encoding idioms appropriate to the active stakeholders of the initiative.

Cross disciplinary focus. Cross-disciplinary discovery is a major founding principle of DCMI and remains the primary focus of the initiative. Promoting interoperability and adoption of DCMI metadata across disciplines often requires refinements and extensions to basic metadata sets. DCMI promotes and facilitates such activities in the interest of increasing the likelihood of interoperability and coherence in the realm of metadata.

Working Groups

DCMI Working Groups are constituted to address work items raised in the course of DCMI activities. Membership of DCMI Working Groups is open to all interested parties. Chairs and Co-Chairs (if any) of Working Groups are appointed by the DCMI Directorate. Working Group activities are conducted through a combination of face-to-face meetings, teleconferencing, and an archived mailing list.

Because the annual DCMI Conference and Workshop provides an opportunity to discuss and review work items and to introduce new people, all working groups automatically expire at each annual event, although they may be re-constituted if necessary.

A set of "Guidelines for Dublin Core Working Groups" is available that define the procedures that are employed in developing Dublin Core specifications and in carrying out other tasks to advance the Dublin Core Metadata Initiative.

Approval Process

The approval processes related to Dublin Core specifications are described in the "Procedure for approval of DCMI Metadata Terms and Recommendations". Procedures exist for semantic proposals that are evaluated by the DCMI Usage Board and for technical proposals that are evaluated by the DCMI Advisory Board. Both procedures include a Public Comment period for all proposals.

3.5 Annual conference and workshop

Since 1995, the Dublin Core Workshop Series has attracted experts from numerous information-based communities to participate in the evolution and maintenance of Dublin Core metadata. The goal of the DCMI Workshops is to promote consensus within the larger metadata community while incorporating the combined wisdom and collective experience of the DCMI's membership into the design of metadata elements, related refinements, and discipline-specific extensions, all within a coherent architecture that supports world-wide information discovery.

In 2001, the Workshop format was broadened to include a tutorial track and a conference paper/poster track in addition to the working group meetings that characterized the earlier workshops. Currently the venue for the annual conference is rotated among North America, Europe, and Austral-Asia to promote worldwide participation.

4 Considerations for the use of Dublin Core

4.1 Discovery and description

Metadata (information about information) can be used to describe information from various perspectives, e.g. to support archiving, to protect intellectual property rights or to manage change of resources.

Rather than addressing all the possible perspectives, the Dublin Core Metadata Initiative developed the Dublin Core Metadata Element Set from a background of discovery requirements. Therefore, the metadata terms maintained by DCMI contain only elements that are useful for discovery of resources. Moreover, it addresses the discovery needs in a cross-domain environment. This indicates that the Dublin Core does not address all description requirements, but only a subset.

Dublin Core metadata can be used in many scenarios. Two examples of the use of Dublin Core metadata are:

Integration of heterogeneous metadata

In some environments, information managed by an organisation or group of organisations is described using various metadata standards. In order to be able to federate and integrate these various types of metadata, Dublin Core may be used as an intermediary format. Conversion from more elaborate formats to Dublin Core is usually reasonably straightforward. However, it needs to be acknowledged that conversion from a detailed description format to the Dublin Core will cause some loss of detail. Because of this, the more elaborate and detailed procedures and data formats will most probably continue to be used while the Dublin Core is then used as the common interchange format.

Simple description mechanism

In less complex environments or where description of resources is started from scratch, Dublin Core may be chosen as the set to describe the resources. Some implementations that do this use only the fifteen core elements, others also use the added specificity offered by the element refinements and encoding schemes. In some cases, additional, local terms can be defined in addition to Dublin Core metadata terms. The declaration of such usage can be done in an Application Profile. Guidelines for Dublin Core Application Profiles are contained in CWA 14855 (ftp://ftp.cenorm.be/PUBLIC/CWAs/e-Europe/MMI-DC/cwa14855-00-2003-Nov.pdf).

4.2 Compliance

In principle, none of the Dublin Core metadata terms are mandatory. As the terms defined in Dublin Core are meant to be useful for a wide range of applications, it would be difficult to prescribe the use of particular terms. For example, not all resources need to have a title and for some resources it is not known or not relevant who created them. Making terms mandatory in all cases would lead to metadata values of ‘not applicable’ or similar which would not be very helpful.

Of course, in specific applications, it may well be necessary to mandate the use of specific Dublin Core elements, refinement of encoding schemes. This type of usage can be declared in Application Profiles. Any combination of elements and element refinements is allowed, as long as their usage adheres to the semantics declared in the base documentation of the Dublin Core metadata terms.

Implementers or specific communities can ask DCMI for review of their Application Profiles against the semantic definitions and grammatical principles. The DCMI Usage Board has defined the criteria and procedures according to which such reviews take place (see: http://dublincore.org/usage/documents/profiles/ and http://www.dublincore.org/usage/documents/process/#six.)

4.3 Interoperability

Cross-domain interoperability is a key aspect of the Dublin Core Metadata Initiative. Intended as a Core set of such commonly understood descriptors, Dublin Core metadata aims to help interoperability across various domains. Adherence to the Core will make it possible for users who are not specifically looking for resources in a particular domain, to find resources that are related to their information needs. Users who are looking for resources in a specific domain will generally have more specific needs, and therefore domain-specific initiatives will continue to have specific requirements for their metadata.

The first level of interoperability is the use of Dublin Core metadata terms, i.e. which elements and element refinements are being used in a particular application. The terms provide a small vocabulary that can be understood across many communities, and act as a lingua franca across applications and domains.

The second level of interoperability is formed by the values that are assigned to the metadata terms, for example the title, the name of the creator, the subject of the content of the resource or the language it is written in. As these values are domain-, application-, and even resource-specific, no strict recommendations can be given that are valid for all implementations. However, some general considerations apply and recommendations on best practice can be given. Both the comment fields in the base specification of the Dublin Core metadata terms, as well as the DCMI User Guide "Using Dublin Core" give a number of suggestions and examples related to the values of the metadata terms (see: http://dublincore.org/documents/dcmi-terms/ and http://dublincore.org/documents/usageguide/).

Examples of these suggestions are:

  • Specify personal names in the form <family name>, <given name>, for example "Smith, John"
  • Use the Date and Time Format defined by the World Wide Web Consortium (W3CDTF) for dates, for example 2004-07-12 for 12 July 2004
  • Use Uniform Resource Identifiers (URI) for identifiers and pointers to other resources
  • Use values from controlled vocabularies or formal classification schemes where appropriate, for example, IANA MIME types for format of the resource, terms from a geospatial thesaurus for the coverage of a resource, or RFC3066 with ISO 639 for the language of a resource.

Adhering to such a set of basic recommendation will increase interoperability across domains.

4.4 Granularity

In various discussions, the question has been raised: "What is a resource?" As the answer might be different in different communities and different projects, it seems that a broadly acceptable answer might be: "Anything that you want to be discovered". For specific situations, the type of the resource could range from "sets or collections of things" to "parts of things". This can be referred to as the granularity of description.

Dublin Core implementations do indeed cover a wide range of resources with a variety of granularities, including sets or collections of objects, individual objects, as well as the separate parts of objects.

Examples:

  • A Web site as a collection of pages or the complete set of information held by an organisation
  • A single Web page, or a single document or image
  • An image that is part of a slide show

4.5 Embedded vs. separate metadata

Metadata can be embedded in the resource it describes (for example in META tags in HTML) or stored separately from the resource (for example in a database). These approaches have different characteristics in terms of the creation, maintenance and harvesting of the metadata.

Current implementations are increasingly using a hybrid approach where metadata is stored and managed in a database or as a set of separate files, and included in the resources when they are published. An example is the DCMI Web site itself, where all metadata is maintained as separate .RDF files. The publication process of the Web site includes a step where the .RDF file is read, converted to META tags, and included in the Web page. See for example the DCMI home page http://dublincore.org/index.shtml that contains a pointer towards the bottom to the metadata file http://dublincore.org/index.shtml.rdf, but also DC metadata embedded in the HTML source.

4.6 Search engines

When we are talking about metadata for discovery of Web resources, one of the mechanisms that need to be taken into consideration is the use of search engines. A search engine is a system that runs a program with a certain regularity to access Web pages and collect information to build indexes that can be used to discover resources. There are two types of search engines. The first type is the general search engine that collects information freely available on the Web and indexes this for a general audience; the second type is the specialised or subject-specific search engine that only gathers information from known sources or specific subjects, usually for a specific audience.

General search engines do not generally use metadata because (a) the percentage of resources with metadata is relatively low; (b) there is no easy way to determine the quality of metadata. Indexes are generally based on full-text of documents on the Web. Some use patented algorithms to derive relevance of resources from external factors such as the number of links to a resource, others employ cataloguers to describe and classify resources. Recently, search engine companies have started to look for partnerships with metadata experts (e.g. in libraries) or providers of high quality metadata (e.g. database hosts) in an attempt to enhance indexing and ranking strategies by using high-quality metadata to increase the quality of search results.

Specialised search engines, on the other hand, aggregate and index metadata associated with managed collections, held in a controlled environment such a company or a set of collaborating organisations, where certain quality criteria and implementation guidelines are being applied. A number of specialised search engines do, indeed, use Dublin Core elements for their indexing. In these services, in addition to simple word or phrase searches, the user can select specific indexes to search for authors, titles, keywords, and subject-specific classification schemes taking advantage of the more detailed descriptions.
