Milestone 1 Alpha 2

This release is very far from feature-complete, but serves as an initial demonstration of the possibility to obtain RDF data from LOC MARC records, using the RDA and FRBR schemas.

These are gzipped n-triples files. Each file contains of the order of 5 million lines (i.e. triples), except part01 which is a bit smaller (~3 million).

There may be a serious encoding issue with these data dumps, see below. These data were obtained via the following steps:

  1. convert marc 21 (marc encoding) to marc xml (UTF-8) via yaz-marcump
  2. split marc xml files into 10,000 record splits via python/bash scripts
  3. convert marc xml to mods xml via MARC21slim2MODS3.xsl
  4. convert mods xml to rdf xml via mods2rda
  5. translate to n-triples and concatenate splits via jena.rdfcat

Features

The following features of the source data have been implemented in the transformation to RDF:

  • for each marc record, a description of a manifestation, an expression and a work are generated, and are linked to each other as appropriate
  • the title and subtitle are used to generate title proper and parallel title of the manifestation, and also preferred title for the work
  • the statement of responsilibity for the title is generated
  • personal names are used to generate descriptions of people, and to generate appropriate links between the work and the persons based on the relator (only creator is implemented currently)
  • lcsh subject topics are used to generate subjects of the work
  • key properties are added for works and persons, based on a very simple hash of the work title or person name respectively, to aid in identifying works and persons across records

Issues

During the final stage, rdfcat reports a substantial number of warnings, reporting that some content is not in unicode normal form C, associated with characters that are accented in some way. This is probably a serious issue in the data, and needs to be resolved prior to any real deployment. I do not know whether this is an artefact of yaz-marcdump or some other stage in the processing chain, and I do not know what the consequences will be for SPARQL queries or text indexes. The data does, however, load into a Jena TDB store without complaint. See also issue 22.

There are also plenty of design issues outstanding, see the issues list for more.

Design and Implementation Method

The following method was used to design and implement the modular xslt stylesheet which computes the transformation from mods xml to rdf.

For each feature...

  1. design expected output of transformation for one or more sample records, e.g. expected record 00000002
  2. implement new module for mods2rda main stylesheet
  3. run transformation on one or more sample records to generate e.g. output record 00000002
  4. perform rdf graph diff between expected and actual output, if no differences then implementation of feature is considered done
